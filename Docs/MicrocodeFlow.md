# Microcode Compilation Flow

This document explains how the microcode in this project is compiled, what files participate in the flow, and how the internal code path works from handwritten instruction tables to final EEPROM binary images.

---

## 1. Purpose

The `Microcode/` directory contains the tooling that converts human-written instruction descriptions into binary ROM images for the three microcode EEPROMs used by the CPU.

The pipeline does **not** directly compile assembly programs. Instead, it compiles the **control logic tables** that define how each CPU instruction behaves cycle by cycle.

At a high level, the flow is:

1. Read the YAML configuration
2. Read handwritten instruction tables from `Microcode/Instructions/`
3. Auto-generate normalized instruction tables in physical chip order
4. Parse those autogenerated tables into address/data matrices
5. Expand wildcard address bits into concrete EEPROM addresses
6. Encode output rows into bytes
7. Write one binary image per microcode chip

---

## 2. Main Files Involved

### Configuration
- `Microcode/MicroCodeConfig.yaml`

Defines:
- supported instruction names
- input control signal encoding
- output control signal encoding
- chip-specific extra signals
- virtual control pins for `uCode1`

### Stage 1: Autogen builder
- `Microcode/GenerateAutogenInstructions.py`

Reads handwritten instruction files and generates normalized instruction modules into:
- `Microcode/out/autogen/`

### Stage 2: Parser and ROM image generator
- `Microcode/CompileAutogenInstructions.py`

Reads autogenerated instruction modules and converts them into:
- EEPROM byte arrays
- address/data map text output

### Pipeline entry point
- `Microcode/GenMicrocode.py`

Runs the full flow end-to-end.

### Handwritten instruction sources
- `Microcode/Instructions/Ins*.py`

Each file contains one instruction table in a Python string named `INS`.

---

## 3. Output Files Produced

When `python3 GenMicrocode.py` is run inside `Microcode/`, the `out/` directory is recreated and populated with:

- `out/autogen/Autogen_<Instruction>.py`
- `out/microCodeMap.txt`
- `out/uCode0.bin`
- `out/uCode1.bin`
- `out/uCode2.bin`

### Meaning of each output

#### `Autogen_<Instruction>.py`
A normalized version of the handwritten instruction table.  
This file is arranged in the exact physical chip grouping expected by the parser.

#### `microCodeMap.txt`
A human-readable address-to-byte map showing what value is written to each EEPROM address.

#### `uCode0.bin`, `uCode1.bin`, `uCode2.bin`
Final binary images for the three microcode EEPROM chips.

## 4. Configuration Model

The YAML file is the root of the microcode definition.

## 4.1 Instruction list

In `MicroCodeConfig.yaml`:

```yaml
InsConfig:
  Instructions:
    - NOP
    - OUT
    - HLT
    ...
```

This list is the authoritative instruction set.  
Instruction files must match this list exactly.

---

## 4.2 Input and output control encoding

```yaml
PinConfig:
  InputControl:
    rAI  : 0b0000
    rBI  : 0b0001
    ...
  OutputControl:
    rAO  : 0b0000
    rBO  : 0b0001
    ...
```

These sections define the logical control signals that are encoded through virtual control pins.

- `InputControl` signals are encoded into `uCode1` input virtual pins
- `OutputControl` signals are encoded into `uCode1` output virtual pins

The encoding width is 4 bits, so each group supports up to 16 logical signals.

---

## 4.3 Microcode chip layout

```yaml
MicrocodeChips:
  uCode0:
    FlSe0: null
    FlSe1: null
    FlSe2: null
  uCode1:
    InputVirCtrlPins:
      ViIn0: null
      ViIn1: null
      ViIn2: null
      ViIn3: null
    OutputVirCtrlPins:
      ViOt0: null
      ViOt1: null
      ViOt2: null
      ViOt3: null
  uCode2:
    PCL  : null
    PCC  : null
    AdSu : null
    HLT  : null
    SpUd : null
    SpC  : null
    Cin  : null
    Sgn  : null
```

This is the physical chip-oriented output layout.

- `uCode0` contains extra direct control signals
- `uCode1` contains virtual encoded control pins
- `uCode2` contains another block of direct control signals

This structure is important because the autogenerated instruction files are emitted in this exact order.

---

## 5. Handwritten Instruction File Format

Each instruction file in `Microcode/Instructions/` contains a Python string:

```python
INS = '''
INSTRUCTION:    ADD
...
'''
```

The table contains:
- input rows (`I`)
- output rows (`O`)
- separators and headers

### Important rule
Every instruction file must declare **every configured signal row**, even if unused.

That means:
- all configured input control signals must appear
- all configured output control signals must appear
- all configured extra signals must appear

Unused rows are still required so every instruction table stays complete and uniform.

## 6. Full Pipeline Walkthrough

## 6.1 Entry point: `GenMicrocode.py`

File:
- `Microcode/GenMicrocode.py`

Main flow:

```python
autoGen = GenerateAutogenInstructions.GenAutoInstructions()
autoGen.AutogenEachInstruction()

insParser = CompileAutogenInstructions.ParseInstructions()
insParser.ParseEachInstruction()
generationResult = insParser.GenerateAddressDataMap()
```

This means the pipeline has two major phases:

1. **Autogen phase**
2. **Parse + ROM generation phase**

Before running, the script deletes the old `out/` directory.

---

## 7. Phase 1: Autogen Instruction Generation

Handled by:
- `Microcode/GenerateAutogenInstructions.py`

Class:
- `GenAutoInstructions`

### 7.1 Constructor behavior

In `__init__()`:

- loads `MicroCodeConfig.yaml`
- dynamically imports all `Instructions/Ins*.py` modules
- sorts them
- validates config structure
- validates instruction module consistency

### 7.2 Config validation

`validateConfigStructure()` checks:

- `uCode1` contains `InputVirCtrlPins`
- `uCode1` contains `OutputVirCtrlPins`
- each virtual pin group has exactly 4 pins
- there are no duplicate configured signals

### 7.3 Instruction module validation

`validateInstructionModuleConsistency()` checks:

- every instruction module has an `INS` string
- every module declares `INSTRUCTION: <name>`
- file name matches declared instruction name
- no duplicate instruction declarations exist
- every discovered instruction is listed in YAML
- every YAML instruction has a corresponding file

This keeps the instruction set deterministic.

---

## 8. Parsing Handwritten Instruction Rows

Method:
- `parseInstructionRows()`

This method reads the raw instruction table and separates rows into buckets:

- `OtherLines`
- `InputSignalRows`
- `OutputSignalRows`
- `ExtraSignalLinesByChip`

### How rows are classified

For output rows:
- if the signal belongs to `Extra`, it is stored separately
- if the signal belongs to `OutputControl`, it goes into output-control rows
- if the signal belongs to `InputControl`, it goes into input-control rows

This may look unusual, but it matches the project’s control encoding model:
some rows written as `O` in the source table are later encoded into virtual input-control or output-control pin groups.

### Completeness enforcement

`validateInstructionSignalCompleteness()` ensures every configured signal row exists in every instruction file.

If any signal is missing, generation stops with an error.

---

## 9. Determining Instruction Width

Method:
- `getInstructionColumnCount()`

This determines how many micro-timing columns an instruction has.

It checks the first available row from:
- input rows
- output rows
- `uCode0` extra rows
- `uCode2` extra rows

The returned width is used for:
- virtual signal generation
- filler row generation
- autogen output formatting

---

## 10. Virtual Signal Generation

Methods:
- `autogenInputSignalLines()`
- `autogenOutputSignalLines()`
- shared helper: `autogenVirtualSignalLines()`

This is one of the most important parts of the flow.

## 10.1 Why virtual signals exist

The hardware does not dedicate one EEPROM output bit to every logical control signal.

Instead:
- logical input-control signals are encoded into 4 virtual pins
- logical output-control signals are encoded into another 4 virtual pins

So instead of directly storing many control lines in ROM, the ROM stores a 4-bit selector.

## 10.2 How encoding works

For each micro-timing column:

1. scan all rows in the relevant signal group
2. find which logical signal is active (`1`)
3. look up its configured numeric index from YAML
4. split that index into 4 bits
5. append those bits to the virtual pin rows

If no signal is active in that column, all virtual pins default to `1`.

### Important recent fix
The generator now ensures every virtual row spans the **full instruction width**.  
Previously, virtual rows could become shorter when `-` cells were encountered incorrectly. That issue is now fixed.

### Width validation
`validateGeneratedSignalWidths()` verifies that every generated virtual row has exactly the same number of columns as the instruction.

---

## 11. Writing Autogenerated Instruction Files

Method:
- `createAutoGenInstructionFile()`

This writes:
- `out/autogen/Autogen_<Instruction>.py`

The emitted output is arranged in physical EEPROM order:

1. original non-output lines and input/address rows
2. `uCode0` rows
3. filler rows to make `uCode0` exactly 8 rows
4. separator
5. `uCode1` virtual input rows
6. `uCode1` virtual output rows
7. separator
8. `uCode2` rows
9. filler rows to make `uCode2` exactly 8 rows
10. final footer lines

### Why this matters
`CompileAutogenInstructions.py` assumes:
- 15 input/address rows
- then 8 rows for `uCode0`
- then 8 rows for `uCode1`
- then 8 rows for `uCode2`

So the autogen stage is effectively a normalization stage that converts handwritten instruction tables into a parser-friendly physical layout.

---

## 12. Phase 2: Parsing Autogenerated Instructions

Handled by:
- `Microcode/CompileAutogenInstructions.py`

Class:
- `ParseInstructions`

### 12.1 Constructor behavior

In `__init__()`:

- loads all `out/autogen/Autogen_*.py` modules
- sorts them
- prepares `instructionParsedData`

---

## 13. Parsing the Autogen Tables

Method:
- `ParseEachInstruction()`

For each autogenerated instruction:

1. read the `INS` string
2. extract the instruction name
3. parse all `I` rows into `AddressMatrix`
4. parse all `O` rows into chip-grouped output matrices

### Expected structure

The parser expects:

- 15 input rows total
- 24 output rows total
- 8 output rows per chip

Constants used:

- `EXPECTED_INPUT_ADDRESS_ROW_COUNT = 15`
- `OUTPUT_ROWS_PER_CHIP = 8`

If the shape does not match, generation fails.

---

## 14. PCB-Specific Input Row Remapping

Method:
- `remapInstructionInputRowsForPcbLayout()`

The input/address matrix contains 15 rows total.

Only the **last 8 rows** correspond to instruction register bits that need PCB-specific remapping.

So the parser:
- keeps the first 7 rows unchanged
- reverses only the final 8 rows

This remapping is applied to:
- `MicroInstructionMatrix.InputSignals`
- `AddressMatrix`

This is important because the physical PCB wiring order differs from the logical source-table order.

---

## 15. Internal Parsed Data Structures

`CompileAutogenInstructions.py` uses typed dataclasses:

### `ParsedMicroInstructionMatrix`
Stores:
- input signal names
- output signal names grouped by chip

### `ParsedInstructionData`
Stores:
- instruction name
- address matrix
- output matrix by chip
- micro-instruction matrix metadata

### `GeneratedMicrocodeResult`
Stores:
- final EEPROM byte arrays by chip
- last parsed micro-instruction matrix

These structures make the pipeline easier to validate and debug.

---

## 16. Address Expansion

Method:
- `expandAddressColumn()`

Each instruction timing column corresponds to one address bit-column across the 15 input rows.

The address bits may contain:
- `0`
- `1`
- wildcard-like symbols such as `-`

The method works like this:

1. reverse the bit order to match physical wiring
2. iterate through all possible values for the bit width
3. force fixed bits where the column contains `0` or `1`
4. allow all combinations where the column contains non-binary symbols
5. collect all resulting concrete addresses

So one timing column may map to many EEPROM addresses.

---

## 17. Data Byte Encoding

Method:
- `encodeDataColumn()`

For each chip and each timing column:

1. take the 8 output bits for that chip
2. reverse them to match physical bit order
3. set a bit in the output byte only where the row value is `1`

The result is one 8-bit EEPROM data value.

---

## 18. Matrix Transposition

Method:
- `transposeMatrix()`

The instruction tables are stored row-major in text form, but ROM generation works column by column.

So the parser transposes:

- address rows -> address columns
- output rows -> data columns

Each column then represents one micro-instruction step.

---

## 19. EEPROM Image Generation

Method:
- `GenerateAddressDataMap()`

This is the final compilation stage.

### Steps performed

1. create empty 32 KB arrays for:
   - `uCode0`
   - `uCode1`
   - `uCode2`

2. recreate `out/microCodeMap.txt`

3. for each instruction:
   - transpose address matrix into columns
   - transpose each chip output matrix into columns

4. for each timing column:
   - expand the address column into all matching EEPROM addresses
   - encode the chip output column into one byte
   - write that byte into every matching address

5. log the mapping into `microCodeMap.txt`

### ROM size
Each chip image is currently created as:

```python
microcodeSize = 32768
```

So each output binary is 32 KB.

---

## 20. Binary File Writing

Back in `GenMicrocode.py`, after generation:

```python
for chipName, eachChipMicrocode in generationResult.MicrocodeByChip.items():
    genMicrocode(chipName, eachChipMicrocode)
```

This writes:

- `out/uCode0.bin`
- `out/uCode1.bin`
- `out/uCode2.bin`

Each file contains the final EEPROM contents for one chip.

## 21. End-to-End Flow Summary

The complete flow is:

### Step 1
`GenMicrocode.py` starts and deletes old `out/`

### Step 2
`GenerateAutogenInstructions.GenAutoInstructions()` loads config and instruction modules

### Step 3
Handwritten instruction tables are validated against YAML

### Step 4
Logical control rows are converted into virtual pin rows for `uCode1`

### Step 5
Normalized autogen instruction files are written in physical chip order

### Step 6
`CompileAutogenInstructions.ParseInstructions()` loads autogen modules

### Step 7
Autogen tables are parsed into:
- address matrix
- per-chip output matrices

### Step 8
Instruction-register input rows are remapped for PCB wiring

### Step 9
Address columns are expanded into concrete EEPROM addresses

### Step 10
Output columns are encoded into bytes

### Step 11
Bytes are written into chip-specific ROM arrays

### Step 12
Final `.bin` files and `microCodeMap.txt` are written

---

## 22. Practical Example

For an instruction like `ADD`:

1. `Instructions/InsADD.py` defines the human-readable table
2. `GenerateAutogenInstructions.py`:
   - validates all required rows exist
   - generates `ViIn0..ViIn3`
   - generates `ViOt0..ViOt3`
   - writes `out/autogen/AutogenADD.py`
3. `CompileAutogenInstructions.py` reads `AutogenADD.py`
4. it splits rows into:
   - 15 address rows
   - 8 rows for `uCode0`
   - 8 rows for `uCode1`
   - 8 rows for `uCode2`
5. each timing column becomes:
   - one set of EEPROM addresses
   - one byte per chip
6. those bytes are written into the final ROM images

## 23. Important Invariants

The current pipeline depends on these rules:

### Instruction source rules
- every instruction file must declare every configured signal row
- instruction file name must match declared instruction name
- every configured instruction must exist as a file

### Shape rules
- exactly 15 input rows in autogen instruction tables
- exactly 8 output rows per chip
- exactly 24 output rows total

### Config rules
- `uCode1` must define both virtual pin groups
- each virtual pin group must contain exactly 4 pins
- no duplicate configured signals

### Generation rules
- all generated virtual rows must match instruction width
- autogen output must remain in physical chip order

- imported instruction count

- creation of `uCode0`, `uCode1`, `uCode2`

## 25. Debugging Tips

If generation fails, check these areas first:

### Missing signal row
Usually caused by a handwritten instruction file not declaring one of the configured signals.

### Instruction mismatch
Check:
- file name
- `INSTRUCTION:` declaration
- YAML instruction list

### Wrong row counts
Check the autogenerated file in:
- `Microcode/out/autogen/`

Make sure it still has:
- 15 input rows
- 8 rows for each chip block

### Wrong virtual row width
Check whether generated `ViIn*` and `ViOt*` rows have the same number of columns as the instruction.

### Wrong chip grouping
Check whether extra signals are still assigned to the correct chip:
- `uCode0`
- `uCode2`

## 26. Suggested Future Improvement

One architectural weakness still remains:

The pipeline transports instruction data through dynamically imported Python modules (`Ins*.py` and `Autogen_*.py`).

This works, but a simpler intermediate format such as:
- JSON
- YAML
- CSV-like structured text

would make:
- validation easier
- debugging easier
- tooling simpler
- accidental import/cache issues less likely

That is a recommended future cleanup, but the current pipeline is now functioning correctly.

## 27. Short Mental Model

If you want one compact mental model, use this:

- `MicroCodeConfig.yaml` defines the legal signals and chip layout
- `Instructions/Ins*.py` defines instruction behavior in human-readable tables
- `GenerateAutogenInstructions.py` converts those tables into normalized physical-chip tables
- `CompileAutogenInstructions.py` converts those tables into EEPROM addresses and bytes
- `GenMicrocode.py` writes the final ROM binaries
